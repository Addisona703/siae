##### 1. (高风险) 安全性问题：Refresh Token 存储在 localStorage

- **问题：** 设计文档明确指出“访问令牌”和“刷新令牌”都存储在 `localStorage`。这是一个重大的安全隐患。
- **风险：** `localStorage` 容易受到 XSS (跨站脚本) 攻击。如果攻击者注入了恶意脚本，他们可以轻松窃取长达 7 天有效期的 `Refresh Token`，从而永久性地劫持用户会话。
- **建议：** `Access Token`（访问令牌）存储在 `localStorage` 或内存中是可以接受的，但 `Refresh Token`（刷新令牌）**强烈建议**由后端通过 `HttpOnly` Cookie 传递。`HttpOnly` Cookie 无法被 JavaScript 访问，可以从根本上防止 XSS 窃取。

##### 2. (中风险) 权限数据的来源不明确

- **问题：** `Permission Store` 需要 `All permissions tree` (完整的权限树) 来渲染 `权限管理` 模块中的树形选择器。但是 `数据模型` 或 `API 客户端架构` 中均未定义用于获取这个“完整权限树”的 API 接口。
- **分析：** `UserInfo` 只包含了用户*拥有*的权限 (`permissions: string[]`)，但这不足以渲染整个树。
- **建议：** 明确设计一个 API 端点（例如 `GET /api/v1/permissions/tree`），用于获取完整的权限树结构。
